# sush.sh (sush) v0.6 需求文档

## 版本目标

v0.6 的目标是为 SSH 登录密码和私钥口令接入系统原生安全存储，在不引入应用级主密码的前提下，减少重复输入并保证凭证不落盘到普通配置文件。

## 变更范围

本版本在 `v0.5` 基础上增强认证流程和本地 metadata，不改变主机管理、SSH / SFTP 主流程。

## 需求细则

### F1: 系统安全存储优先读取

**规则**：
- 登录密码和私钥口令都优先从系统原生安全存储读取
- 平台能力如下：
  - macOS：Keychain
  - Windows：Credential Manager
  - Linux：Secret Service
- 保持认证顺序为：`ssh-agent -> 无口令私钥 -> 系统安全存储 / 手工输入`

### F2: 认证成功后静默保存

**规则**：
- 当用户手工输入登录密码或私钥口令并认证成功后，sush 静默尝试写入系统安全存储
- 保存失败不影响本次已成功连接
- 下次再次需要输入时，优先尝试读取已保存凭证

### F3: 凭证不写入普通配置

**规则**：
- `hosts.toml` 只保存非敏感主机信息
- 不允许把明文密码、私钥口令或其可逆内容写入配置文件
- 本地 metadata 只允许记录“上次保存失败原因”等非敏感提示信息

### F4: Linux 无可用 Secret Service 的降级策略

**规则**：
- 当 Linux 没有可用的 Secret Service 时，禁止保存凭证
- 允许用户本次临时输入并继续连接
- 下次再次需要输入时，应提示上次未保存的原因
- 提示文案应建议安装并解锁 `gnome-keyring` 或 `kwallet`
- 不允许自动回退到本地加密文件或应用级主密码方案

## 验收重点

- 用户首次成功输入后，后续连接尽量无需重复输入相同凭证
- 凭证不会出现在 `hosts.toml` 等普通配置中
- Linux 缺少系统能力时，行为清晰、可继续连接，但不伪造“已保存”能力
