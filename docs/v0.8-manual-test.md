# v0.8 手动测试清单

## 1. 目的

本文用于手动验证 `v0.8` 的核心功能是否可用，先以可执行 checklist 为主，后续再逐步细化。

当前覆盖范围：

- 端口转发管理器基本交互
- `Local / Remote / Dynamic` 三类转发
- 单跳 `ProxyJump`
- daemon 持续运行与状态恢复
- macOS 钥匙串中的 SSH 登录密码保存与复用

当前不覆盖：

- 自动化测试
- 多跳 `ProxyJump`
- Windows 平台行为
- 私钥口令保存到系统钥匙串

## 2. 测试拓扑

### 2.1 机器角色

客户端：

- macOS，运行 `sush`

服务器：

- `ubuntu1`: `ubuntu1.orb.local` / `192.168.139.187`
- `ubuntu2`: `ubuntu2.orb.local` / `192.168.139.169`

建议分工：

- `ubuntu1` 作为跳板机、远程转发落点、密码登录测试机
- `ubuntu2` 作为目标机、被访问服务机，仅允许通过 `ubuntu1` 跳转访问

当前访问约束：

- `Mac -> ubuntu1` 使用密码登录，用于验证 macOS 钥匙串保存 SSH 登录密码
- `Mac -> ubuntu2` 不允许直连，应通过 `Mac -> ubuntu1 -> ubuntu2`
- `ubuntu2` 的 SSH 公钥信任链使用 macOS 本地私钥，不使用 `ubuntu1` 本地私钥

### 2.2 预置服务

在 `ubuntu2` 启动一个监听所有网卡地址的 HTTP 服务，供本地转发、动态转发、`ProxyJump` 验证使用：

```bash
python3 -m http.server 18080 --bind 0.0.0.0
```

在 macOS 启动一个仅监听本机回环地址的 HTTP 服务，供远程转发验证使用：

```bash
python3 -m http.server 18081 --bind 127.0.0.1
```

## 3. 前置条件

### 3.1 环境要求

- macOS 能正常构建并运行 `sush`
- 两台 Ubuntu 已开启 SSH 服务
- `ubuntu1` 支持密码登录，用于验证 macOS 钥匙串保存
- `ubuntu2` 已配置为仅允许通过 `ubuntu1` 访问
- 已知两台 Ubuntu 的 SSH 用户名；当前测试按 `root` 用户编排
- 本机没有占用后续测试需要的本地端口：`10080`、`10081`、`1080`

### 3.2 建议准备

- 为两台 Ubuntu 先准备最小 host 配置
- `ubuntu2` 的 host 条目后续需要填 `proxy_jump = "ubuntu1"`
- `ubuntu2` 的身份文件应指向 macOS 本地私钥，而不是跳板机上的私钥
- 如果要验证 daemon 重启后的状态恢复，建议先确认 `~/.config/sush/` 可正常读写

### 3.3 构建与启动

在项目根目录执行：

```bash
cargo run
```

## 5. 基线验证

### T01 直连 SSH

目标：

- 验证 `ubuntu1` 可直连，`ubuntu2` 必须通过跳板访问

步骤：

1. 启动 `sush`
2. 选择 `ubuntu1`
3. 按 `Enter`
4. 成功进入 SSH 后退出
5. 选择 `ubuntu2`
6. 按 `Enter`
7. 观察是否因已配置 `proxy_jump = "ubuntu1"` 而通过跳板进入 `ubuntu2`
8. 成功进入 SSH 后退出

期望：

- `ubuntu1` 可通过密码直接登录
- `ubuntu2` 不走直连，而是通过 `ubuntu1` 跳转进入 SSH 会话
- 退出后能回到主界面

### T02 直连 SFTP

目标：

- 验证 `ubuntu1` 的直连 SFTP 与 `ubuntu2` 的跳板 SFTP 都可用

步骤：

1. 在主界面选择 `ubuntu1`
2. 按 `s`
3. 进入 SFTP 后浏览目录
4. 退出回主界面
5. 对 `ubuntu2` 重复同样操作

期望：

- 两台主机都能进入 SFTP 视图
- `ubuntu2` 的 SFTP 应通过 `ubuntu1` 跳板建立
- 本地 / 远程面板能正常显示
- 退出后能回到主界面

期望：

- 能正常返回主界面

### T04 规则 CRUD

目标：

步骤：

1. 进入转发管理器
2. 选中 `ubuntu1`
3. 新建一条 `Local` 规则
4. 新建一条 `Remote` 规则
5. 新建一条 `Dynamic` 规则
6. 分别编辑名称和端口
7. 删除其中一条规则

- 编辑后列表立即刷新
- 删除后列表立即消失

## 7. 转发功能验证

### T05 Local Forward

目标：

- 验证本地转发把 macOS 本地端口映射到远端目标服务

规则建议：

| Host      | Type    | Local Port | Remote Host | Remote Port |
| --------- | ------- | ---------- | ----------- | ----------- |
| `ubuntu2` | `Local` | `10080`    | `127.0.0.1` | `18080`     |

步骤：

1. 确认 `ubuntu2` 上 `0.0.0.0:18080` 服务已启动，且 `ubuntu1` 可访问 `192.168.139.169:18080`
2. 在转发管理器中选中 `ubuntu2`
3. 新建或编辑为上述 `Local` 规则
4. 在规则列表中按 `Enter` 启动
5. 在 macOS 执行：

```bash
curl http://127.0.0.1:10080
```

期望：

- 规则状态从 `Connecting` 变为 `Running`
- `curl` 能返回 `ubuntu2` 上 HTTP 服务内容

### T06 Remote Forward

目标：

- 验证远程转发把远端端口映射回 macOS 本地服务

规则建议：

| Host      | Type     | Local Port | Remote Host | Remote Port |
| --------- | -------- | ---------- | ----------- | ----------- |
| `ubuntu1` | `Remote` | `18081`    | `127.0.0.1` | `10081`     |

说明：

- `Local Port` 表示 macOS 上实际被访问的本地端口
- `Remote Host + Remote Port` 表示远端 SSH 服务器上开放的监听地址和端口

步骤：

1. 确认 macOS 上 `127.0.0.1:18081` 服务已启动
2. 在转发管理器中选中 `ubuntu1`
3. 新建或编辑为上述 `Remote` 规则
4. 启动规则
5. 登录 `ubuntu1`
6. 在 `ubuntu1` 执行：

```bash
curl http://127.0.0.1:10081
```

期望：

- 规则状态从 `Connecting` 变为 `Running`
- `ubuntu1` 上访问 `127.0.0.1:10081` 时，实际命中 macOS 上 `18081` 服务

### T07 Dynamic Forward

目标：

- 验证最小 SOCKS5 `CONNECT` 代理可用

规则建议：

| Host      | Type      | Local Port |
| --------- | --------- | ---------- |
| `ubuntu1` | `Dynamic` | `1080`     |

步骤：

1. 确认 `ubuntu2` 上 `127.0.0.1:18080` 服务已启动
2. 在转发管理器中选中 `ubuntu1`
3. 新建或编辑为上述 `Dynamic` 规则
4. 启动规则
5. 在 macOS 执行：

```bash
curl --socks5 127.0.0.1:1080 http://192.168.139.169:18080
```

期望：

- 规则状态变为 `Running`
- 命令能返回 `ubuntu2` 上 HTTP 服务内容

## 11. macOS 钥匙串验证

### K01 首次保存登录密码

目标：

- 验证首次手输 SSH 密码后，成功写入 macOS Keychain

前置条件：

- `ubuntu1` 支持密码登录
- 当前该 host 在 macOS 钥匙串中还没有旧密码条目

步骤：

1. 启动 `sush`
2. 选择 `ubuntu1`
3. 按 `Enter`
4. 手动输入 SSH 登录密码
5. 成功进入 SSH 后退出

期望：

- 首次连接成功
- 认证成功后密码被写入系统钥匙串

可选检查：

当前测试按 `ubuntu1` 作为 host alias：

```bash
security find-generic-password -s sush -a 'ubuntu1:login_password' >/dev/null && echo found
```

### K02 二次连接复用密码

目标：

- 验证第二次连接时直接复用系统钥匙串中的密码

步骤：

1. 完成 `K01`
2. 再次选择同一 host
3. 按 `Enter`

期望：

- 不再弹出密码输入框
- 能直接进入 SSH

### K03 SFTP 复用密码

目标：

- 验证 SFTP 路径也会复用已保存的登录密码

步骤：

1. 完成 `K01`
2. 回到主界面
3. 选择同一 host
4. 按 `s`

期望：

- 不再要求重新输入密码
- 能直接进入 SFTP

### K04 拒绝保存后的提示

目标：

- 验证系统拒绝保存后，下次输入框会展示失败原因

步骤（真实路径）：

1. 第一次连接时，在 macOS 系统弹窗中拒绝写入钥匙串
2. 完成当次连接
3. 断开并再次连接同一 host
4. 观察密码输入框提示

替代步骤（稳定复现）：

1. 退出 `sush`
2. 打开 `~/.config/sush/hosts.toml`
3. 找到目标 host 的 `id`
4. 在 `[metadata]` 下写入一条失败记录：

```toml
secret_save_failures = [
  { account = "<host_id>:login_password", reason = "permission denied by system keyring" },
]
```

1. 启动 `sush`
2. 再次连接同一 host
3. 观察密码输入框提示

说明：

- `account` 使用 host 的内部 `id`，不是 alias
- `login_password` 对应登录密码条目
- 如果 `[metadata]` 下已经存在 `secret_save_failures`，可直接追加或临时只保留这一条

期望：

- 再次连接时仍需要输入密码
- 输入框标题或提示中包含 `Password was not saved last time`
- 输入框标题或提示中包含上次保存失败原因，例如 `permission denied by system keyring`
